网页请求流程

从图中，可知，服务器和客户端的基本交互流程如下：

  1. 用户端通过浏览器发起请求和附带请求数据
  2. 服务器接收并解析请求，交给相应的Servlet类处理
  3. Servlet处理完毕后，将相应的响应数据和响应页面返回给客户端
  4. 客户端浏览器解析响应数据和响应页面，并显示给用户

GET和POST请求格式及响应数据格式

服务器如何解析请求

服务器接收到用户请求后，先会知道该次请求是GET请求还是POST请求，然后以对应请求的方式解析请求数据。比如对于url为http://主机地址:端口号/工程名/映射路径或文件如果为文件，则服务器会直接将文件以响应数据格式传回给用户，通常文件为.html或.jsp。而如果为映射路径的话，yin那么服务器就会解析web.xml文件，根据映射路径找到对应的Servlet类。

服务器在解析用户请求时，会将请求数据以键值对的方式读取和保存，并将数据封装成HttpServletRequest对象供程序使用，并提供方法获取该对象中的数据。同时也会提供HttpServletResponse对象，用于设置相应数据和响应内容。

请求转发和重定向

请求转发：由服务器内部多个servlet进行请求处理，共有同一个请求对象，并可以通过请求对象中的setAttribute添加信息，方便下一个servlet处理。

重定向：接收请求的servlet处理不了用户请求，但知道哪一个资源可以处理请求，通过设置请求头，并将该资源url返回给客户端，让客户端重新发一次新的请求到资源地址，完成请求处理。

Session和Cookie

Cookie

Cookie为服务器通知客户端浏览器保存键值对的一种技术，主要是由于HTTP为无状态协议，每次请求不能够主动携带数据，影响用户某些功能的体验，比如购物车。

Session

客户端和服务器间的会话，一次会话的结束为客户端关闭浏览器。服务器会为每个客户端创建一个会话，并分配一个唯一的id，用于标识存储的用户信息。

Cookie和Session的关联，以及Session如何工作

大体步骤如下：

  1. 解析Request中的Parameters，拿到Session ID并设置到Request对象中
  2. 如果支持Cookie，则从Cookie中拿到Session ID并覆盖第一步中的ID
  3. 根据Session ID在sessions集合中查找已经存在的Session对象，如果存在则返回，不存在则创建一个新的Session对象(Session失效)并添加到sessions集合中
  4. 根据返回的Session对象新增一个Cookie，并将这个Cookie设置到响应头中
  5. 对Session对象进行操作，如修改和删除等

从Request中获取的Session对象保存在org.apache.catalina.Manager类中，其实现类为org.apache.catalina.session.StandardManager。我们可以通过SessionId从StandardManager的sessions集合中取出StandardSession对象。综上，我们可以知道，StandardManager是帮助我们管理StandardSession对象的一个类。

StandardManager类负责Servlet容器中所有StandardSession对象生命周期的管理。当Servlet容器重启或关闭时，StandardManager负责持久化没有过期的StandardSession对象，将对象保存到”SESSIONS.ser”为文件名的文件中。当Servlet容器重启时，它会重新读取这个文件并解析出所有的Session对象，重新保存在StandardManager的sessions集合中。
